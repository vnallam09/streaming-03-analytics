## Phase 4: Technical Modification

**File:** `src/streaming/kafka_consumer_venkat.py`

**Change:** Summarize `profit_estimate` (estimated 30% margin on subtotal) instead of `total` revenue.

The Dr. Case's consumer tracks running stats on the full order `total` (subtotal + tax).
The modified consumer tracks running stats on `profit_estimate`, which is 30% of subtotal.
This shifts focus from gross revenue to estimated profit per order.

The change touches `process_message()` — instead of `stats.update(enriched["total"])`,
the consumer calls `stats.update(profit_estimate)` after computing the profit estimate.

## Phase 5: Apply the Skills

**Scenario:** Profit and category insights from the streaming sales pipeline.

The Dr. Case's consumer validates messages and tracks revenue.
The venkat consumer extends this by adding two derived fields that answer a different question:
*Which product categories are generating the most estimated profit per order?*

### What Changed from the Dr. Case's Example

| Area | Dr. Case's | Venkat |
| --- | --- | --- |
| Reference data loaded | regions.csv only | regions.csv + products.csv |
| Output CSV | `consumed_sales.csv` | `teja_consumed_sales.csv` |
| New derived fields | none | `category`, `profit_estimate` |
| Running stats track | `total` (gross revenue) | `profit_estimate` (30% margin) |

### What the Consumer Does to Each Message

1. Validates required fields against the data contract (unchanged).
2. Enriches with `subtotal`, `tax_amount`, `total` via `enrich_message()` (unchanged).
3. Looks up `category` from `products.csv` using `product_id` as the key.
4. Computes `profit_estimate = subtotal × 0.30` using `compute_profit_estimate()`.
5. Updates running stats on `profit_estimate`.
6. Writes all fields including `category` and `profit_estimate` to the output CSV.

### Derived Fields and Statistics

- **`category`** — product category (e.g., "Streaming", "Data Analytics") from `products.csv`.
  Shows which type of course was purchased, enabling category-level analysis.

- **`profit_estimate`** — subtotal × 30% margin.
  Approximates profit contribution per order without needing actual cost data.

The final summary logs:

- Total estimated profit across all consumed orders
- Average, minimum, and maximum profit per order

### What I Learned from the Stream

Loading a second reference table (`products.csv`) required changing `load_reference_data()`
to return a tuple instead of a single dict, and threading `product_lookup` through
`consume_messages()` and `process_message()`.
This shows how consumer complexity grows naturally as more enrichment lookups are added —
each new reference table adds a parameter to pass through the call chain.

The `compute_profit_estimate` function was added to `derived_fields.py` following the
same pattern as `compute_tax_amount`, keeping the math isolated and testable.
Tracking profit instead of revenue also shows how a small change to one `stats.update()`
call can shift the entire analytics focus of the consumer.
